# 系统总览文档写作方法论

---

## 核心理念

一份好的总览文档回答三个问题：

1. **这个系统存在是为了解决什么问题？**
2. **它的组成部分是什么，彼此如何协作？**
3. **用它的人需要知道哪些非直觉的设计决策？**

不要照搬固定模板。根据系统的实际特点决定用哪些视角、多少篇幅。

---

## 执行流程

### 第一步：提炼核心问题

在写任何内容之前，用一句话定义这个系统的存在价值：

> **当 [角色/场景] 需要 [目标] 时，如何 [解决什么矛盾/难题]？**

这句话会成为文档的"北极星"，之后所有章节都服务于回答它。

### 第二步：画出组件全貌

用 ASCII 树或框图列出所有组件，标注每个组件的单句职责。目的不是美观，是**让读者在 30 秒内看懂系统边界**。

### 第三步：选择视角

根据系统特点，从以下视角中选择 3-5 个，不必全选：

| 视角 | 适用场景 | 常用表达形式 |
|------|---------|------------|
| **组件关系** | 组件多、依赖复杂 | ASCII 框图、层次图 |
| **时序流程** | 有明显的调用顺序、异步交互 | Mermaid sequenceDiagram |
| **状态机** | 有生命周期、状态转换 | ASCII 状态转移图 |
| **决策树** | 使用者需要选择工具/路径 | ASCII 树形决策流 |
| **数据流** | 数据在组件间流转、格式重要 | 线性流程图 + 格式示例 |
| **权限矩阵** | 不同场景下可用性不同 | 表格 |
| **集成点** | 与外部系统有 Hook/事件接口 | 表格 + 配置示例 |

### 第四步：写设计原则

最后提炼 3-5 条设计原则。要求：

- 用**通用的软件工程术语**命名（Fire-and-Observe、CQS、幂等写入……）
- 每条原则必须落地到系统的具体机制，不能泛泛而谈
- 原则之间不重叠

---

## 写作标准

**层次感**：文档应该从"为什么存在"→"长什么样"→"怎么运作"→"怎么用"递进，不要开头就堆工具列表。

**非直觉优先**：重点解释那些"看源码也不容易立刻明白"的决策——为什么用这个格式、为什么这个工具在这个场景被禁用、为什么选协作式取消而不是强制 kill。

**图文结合**：每一张图（状态机、时序图、决策树）之后跟一句 `**关键机制：**`，点名图中最重要的设计。图是眼睛看的，这句话是脑子记的。

**避免**：
- 把所有工具的 API 文档堆在总览里（那是各工具独立文档的事）
- 用"目前"、"接下来"等时态模糊词
- 重复使用相同的 ASCII 图
